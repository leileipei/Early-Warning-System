# SQL 预警系统设计

日期：2026-06-29

## 背景

本项目从空仓库开始，目标是建设一个独立的信息系统预警平台。管理员可以在系统中配置 SQL 预警规则，系统按定时任务或手动触发执行 SQL Server 查询，并将查询到的预警结果通过公司 SMTP 邮箱服务器发送给固定收件人。

## 已确认需求

- 系统形态：独立预警系统，不嵌入现有信息系统。
- 业务数据源：SQL Server。
- 触发语义：SQL 返回的每一行都是一条预警明细。
- 收件人：每条规则固定配置一组收件人。
- 调度方式：支持 Cron 定时执行，也支持管理员手动测试执行。
- 登录权限：第一版提供用户名密码登录，只有管理员可以配置规则。
- 邮件发送：使用公司 SMTP 邮箱服务器。
- 技术栈：Python / FastAPI + 前端管理页面。
- 系统自身存储：SQLite，保存用户、连接、规则、执行记录和邮件日志。
- 邮件组织：每条规则配置发送模式，支持汇总一封邮件和每行一封邮件。
- 重复发送：第一版不做去重控制，每次执行查到结果就发送。
- SQL 安全：第一版只允许单条 `SELECT` 查询。

## 非目标

第一版不实现以下能力：

- SSO、LDAP、AD 登录集成。
- 复杂角色权限体系。
- Redis、队列化任务分发、多 Worker 横向扩展。
- 重复预警抑制、恢复通知、升级通知。
- 按部门、角色、用户组动态匹配收件人。
- 存储过程或任意 SQL 执行。

这些能力可以在后续版本基于当前模块边界继续扩展。

## 推荐架构

采用 Web + Worker 单体架构：一个代码库，两个运行入口。

- FastAPI Web/API：负责认证、规则配置、手动执行请求、执行记录和邮件日志查询。
- Scheduler/Worker：负责加载启用规则的 Cron 表达式，执行 SQL Server 查询，渲染邮件，通过 SMTP 发送，并写入日志。
- SQLite：保存系统配置和运行记录。
- SQL Server：业务只读数据源。
- SMTP：公司邮件发送服务。

这个方案比单进程内置调度更清晰，也比队列化服务更轻。第一版部署成本低，后续规则数量变多时，可以把 Worker 扩展为队列消费者。

## 核心模块

### 认证与管理员

系统提供用户名密码登录。第一版只有管理员角色。管理员可以配置 SQL Server 连接、SMTP、预警规则，并查看执行记录和邮件日志。

密码使用哈希保存。登录成功后使用会话或令牌维持登录状态。所有管理接口都需要认证。

### SQL Server 数据源配置

管理员可以配置一个或多个 SQL Server 数据源：

- 名称。
- 主机、端口、数据库名。
- 用户名。
- 密码。
- 连接超时。
- 是否启用。

密码加密保存。页面和日志不展示密码明文。推荐使用 SQL Server 只读账号，应用层 SQL 校验和数据库权限共同限制写入风险。

### SMTP 配置

管理员配置公司 SMTP 邮件服务器：

- 主机、端口。
- 是否启用 TLS/SSL。
- 用户名和密码。
- 默认发件人。
- 连接超时。

SMTP 密码加密保存。系统提供测试发送能力，用于验证配置是否可用。

### 预警规则管理

每条预警规则包含：

- 规则名称。
- 是否启用。
- SQL Server 数据源。
- 单条 `SELECT` SQL。
- Cron 表达式。
- 固定收件人 `to` 和可选 `cc`。
- 邮件主题模板。
- 邮件正文模板。
- 发送模式：`summary` 或 `per_row`。
- 查询超时。
- 最大返回行数。
- 备注。

邮件模板支持引用 SQL 返回字段，例如 `{{customer_name}}`、`{{amount}}`。字段缺失或模板渲染失败时，本次规则执行失败，不发送邮件，并写入错误日志。

### 规则执行器

执行器接收规则和触发方式，流程如下：

1. 创建执行记录，状态为运行中。
2. 校验 SQL 只允许单条 `SELECT`。
3. 连接 SQL Server。
4. 执行查询，并应用查询超时和最大返回行数。
5. 查询无结果时，记录成功，发送数量为 0。
6. 查询有结果时，按发送模式生成邮件。
7. 调用 SMTP 发送。
8. 写入执行结果和邮件日志。

触发方式包括：

- `scheduled`：Worker 根据 Cron 自动触发。
- `manual`：管理员从页面手动执行。

### 邮件生成

`summary` 模式：

- 一次规则执行只发送一封邮件。
- 邮件正文包含 SQL 返回结果表格。
- 适合日报、异常清单、批量数据提醒。

`per_row` 模式：

- SQL 返回的每一行生成一封邮件。
- 邮件主题和正文都可以引用当前行字段。
- 适合逐客户、逐工单、逐设备的预警。

第一版固定收件人来自规则配置，不从 SQL 结果字段动态取邮箱。

### 执行记录与邮件日志

执行记录字段包括：

- 规则 ID。
- 触发方式。
- 开始时间、结束时间。
- 状态：`success`、`failed`、`partial_failed`。
- 返回行数。
- 邮件数量。
- 耗时。
- 错误类型。
- 错误摘要。

邮件日志字段包括：

- 执行记录 ID。
- 收件人、抄送人。
- 邮件主题。
- 发送状态。
- SMTP 错误摘要。
- 发送时间。

部分邮件发送失败时，执行记录状态为 `partial_failed`，并在邮件日志中保存每封邮件的发送结果。

## 页面范围

第一版页面包括：

- 登录页。
- 仪表盘：展示最近执行、失败规则和邮件发送概览。
- 规则列表：查看、启用、停用、手动执行。
- 规则新建和编辑页。
- SQL Server 数据源配置页。
- SMTP 配置页。
- 执行记录列表和详情页。
- 邮件发送日志页。

页面以管理效率为主，保持信息密度清晰，避免营销式展示。

## 错误处理

- SQL 校验失败：拒绝保存或拒绝执行，并提示只允许 `SELECT`。
- SQL Server 连接失败：本次执行失败，记录连接错误摘要。
- SQL 查询超时：本次执行失败，记录超时。
- SQL 返回行数超过限制：截断到最大行数，记录实际处理行数和截断标记。
- 模板渲染失败：本次执行失败，不发送邮件。
- SMTP 全部失败：本次执行失败，记录 SMTP 错误。
- SMTP 部分失败：本次执行部分失败，逐封记录状态。
- 手动执行：页面展示即时执行结果，包括返回行数、发送数量和错误摘要。

## 安全边界

- SQL 只允许单条 `SELECT`。
- 拦截 `INSERT`、`UPDATE`、`DELETE`、`MERGE`、`DROP`、`ALTER`、`CREATE`、`TRUNCATE`、`EXEC` 等危险语句。
- 拦截多语句执行。
- 推荐 SQL Server 使用只读账号。
- SQL Server 密码和 SMTP 密码加密保存。
- 密码不出现在接口响应、页面明文或日志中。
- 所有管理接口要求登录。

SQL 校验用于降低误操作风险，但不能替代数据库权限。生产部署时必须使用最小权限数据库账号。

## 测试方案

第一版测试覆盖以下方面：

- SQL 校验：允许合法 `SELECT`，拒绝写入、DDL、存储过程和多语句。
- 规则执行：无结果、有结果、查询超时、数据库连接失败。
- 邮件生成：`summary` 表格邮件和 `per_row` 单行邮件。
- SMTP 发送：全部成功、全部失败、部分失败。
- API：登录、规则 CRUD、手动执行、执行记录查询、邮件日志查询。
- Worker：Cron 规则加载、启用/停用规则变化、定时执行。

外部 SQL Server 和 SMTP 在测试中使用适配器或桩对象替代，避免测试依赖真实外部服务。

## 成功标准

第一版完成后，管理员可以：

1. 登录系统。
2. 配置 SQL Server 数据源。
3. 配置 SMTP 邮件服务器。
4. 创建一条只读 `SELECT` 预警规则。
5. 选择 `summary` 或 `per_row` 发送模式。
6. 手动测试规则。
7. 启用 Cron 定时执行。
8. 当 SQL 返回预警行时收到邮件。
9. 在页面查看执行记录和邮件发送日志。

## 后续演进

后续可以按优先级扩展：

- 重复预警抑制和恢复通知。
- 按 SQL 字段动态决定收件人。
- 用户组、部门、角色收件人目录。
- SSO/LDAP/AD。
- PostgreSQL 或 SQL Server 作为系统自身存储。
- Redis 队列和多 Worker。
- 规则版本历史和审批流程。

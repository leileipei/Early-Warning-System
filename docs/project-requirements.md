# 信息系统 SQL 预警系统项目需求

## 1. 项目背景

现有信息系统中的业务数据存储在 SQL Server 中。业务人员或管理员需要通过编写 SQL 查询识别异常数据、风险数据或待处理数据，并在满足预警条件时将结果发送到指定邮箱。

本项目建设一个独立的 SQL 预警系统，提供后台管理页面，用于配置数据库连接、邮件服务、预警规则、调度周期和邮件模板。系统按规则定时执行 SQL，也支持管理员手动执行测试，并记录执行结果和邮件发送日志。

## 2. 项目目标

- 支持管理员通过页面配置 SQL Server 数据源。
- 支持管理员配置 SMTP 邮件服务。
- 支持通过 SQL 查询定义预警条件。
- 支持 Cron 表达式定时执行预警规则。
- 支持手动执行规则，便于测试和排查。
- 支持将 SQL 查询结果发送到规则配置的邮箱。
- 支持查看执行记录、失败原因和邮件发送日志。
- 支持 SQL 安全校验和 SQL Server 语法检测，降低误操作风险。

## 3. 用户角色

### 管理员

管理员可以登录系统并完成以下操作：

- 管理 SQL Server 数据源。
- 管理 SMTP 邮件配置。
- 新建、编辑、启用、停用预警规则。
- 手动执行预警规则。
- 查看规则执行日志和邮件发送日志。

第一版系统仅包含管理员角色，不设计复杂权限体系。

## 4. 功能需求

### 4.1 登录认证

- 系统提供登录页面。
- 管理员使用用户名和密码登录。
- 未登录用户不能访问规则、配置、日志等管理页面。
- 管理员密码使用哈希保存。
- 登录会话由系统维护，支持退出登录。

### 4.2 SQL Server 数据源配置

系统应支持配置一个或多个 SQL Server 数据源。

配置字段包括：

- 数据源名称。
- 主机地址。
- 端口，默认 `1433`。
- 数据库名称。
- 用户名。
- 密码。
- 连接超时时间。
- ODBC 驱动名称。
- 服务器地址覆盖值，支持实例名或特殊连接地址。
- Encrypt。
- TrustServerCertificate。
- 额外连接参数。
- 是否启用。

要求：

- 数据源名称不能重复。
- 密码加密保存。
- 页面不展示密码明文。
- 编辑数据源时，如果密码留空，则保留原密码。
- 规则只能关联已存在的数据源。
- 数据源列表应提供连接测试入口，用于验证已保存配置是否可以连接 SQL Server。
- 生产环境推荐使用 SQL Server 只读账号。

### 4.3 SMTP 邮件配置

系统应支持配置 SMTP 邮件服务。

配置字段包括：

- SMTP 服务器地址。
- SMTP 端口。
- 用户名。
- 密码。
- 发件人邮箱。
- TLS 开关。
- SSL 开关。
- 超时时间。
- 是否启用。

要求：

- SMTP 密码加密保存。
- 页面不展示密码明文。
- 规则执行时使用启用状态的 SMTP 配置发送邮件。
- SMTP 配置列表应提供测试发送入口，用于验证已保存配置是否可以发送邮件。

### 4.4 预警规则管理

系统应支持新建和编辑预警规则。

规则字段包括：

- 规则名称。
- 关联 SQL Server 数据源。
- SQL 查询语句。
- Cron 表达式。
- 收件人。
- 抄送人。
- 邮件主题模板。
- 邮件正文模板。
- 发送方式。
- 查询超时时间。
- 最大返回行数。
- 是否启用。

发送方式包括：

- 汇总一封：一次执行发送一封邮件，邮件正文包含查询结果表格。
- 每行一封：SQL 返回的每一行生成一封邮件。

要求：

- SQL 查询不能为空。
- Cron 表达式必须有效。
- 收件人不能为空。
- 规则保存时必须进行 SQL 安全校验。
- 规则可以启用或停用。
- 停用规则不会被定时调度执行。

### 4.5 SQL 检测

规则编辑页面应提供“检测 SQL”按钮。

检测流程：

1. 管理员选择数据源并输入 SQL。
2. 系统先做本地安全校验。
3. 本地安全校验通过后，系统连接所选 SQL Server。
4. 系统让 SQL Server 解析 SQL 语法，但不执行查询。
5. 系统将检测结果显示在页面上。

本地安全校验要求：

- SQL 不能为空。
- 只允许单条 `SELECT` 或 `WITH` 查询。
- 不允许多语句执行。
- 禁止 `insert`、`update`、`delete`、`merge`、`drop`、`alter`、`create`、`truncate`、`exec`、`execute` 等危险关键字。

SQL Server 语法检测要求：

- 必须选择有效数据源。
- 数据源连接失败时，应提示连接失败或 SQL Server 检测失败。
- SQL 语法错误时，应显示 SQL Server 返回的错误信息。
- 语法检测通过时，应提示“SQL Server 语法检测通过”。
- 检测 SQL 不应真实执行查询或修改数据。

### 4.6 规则执行

规则执行方式包括：

- 定时执行：Worker 根据 Cron 表达式自动执行启用规则。
- 手动执行：管理员在规则列表中点击手动执行。

执行流程：

1. 校验规则和数据源配置。
2. 校验 SQL 只允许只读查询。
3. 连接 SQL Server。
4. 执行 SQL 查询。
5. 按最大行数限制读取结果。
6. 如果结果为空，记录执行成功，邮件数量为 0。
7. 如果结果不为空，根据发送方式生成邮件。
8. 调用 SMTP 发送邮件。
9. 保存执行记录和邮件日志。

要求：

- 查询失败时，本次执行标记为失败。
- 邮件全部发送成功时，本次执行标记为成功。
- 邮件部分发送失败时，本次执行标记为部分失败。
- 邮件全部发送失败时，本次执行标记为失败。
- 执行过程中的错误需要记录错误类型和错误摘要。

### 4.7 邮件模板

系统应支持在邮件主题和正文中使用模板变量。

要求：

- 汇总一封模式下，模板可使用规则名称、行数、结果表格等上下文。
- 每行一封模式下，模板可引用当前 SQL 结果行中的字段。
- 模板渲染失败时，本次规则执行失败，不发送邮件。

### 4.8 执行记录

系统应记录每次规则执行结果。

记录内容包括：

- 规则 ID。
- 触发方式：定时或手动。
- 执行状态。
- 开始时间。
- 结束时间。
- SQL 返回行数。
- 成功发送邮件数量。
- 执行耗时。
- 错误类型。
- 错误信息。

### 4.9 邮件日志

系统应记录每封邮件的发送结果。

记录内容包括：

- 所属执行记录。
- 收件人。
- 抄送人。
- 邮件主题。
- 发送状态。
- 错误信息。
- 发送时间。

### 4.10 管理页面

系统应提供以下页面：

- 登录页。
- 仪表盘。
- 规则列表页。
- 新建规则页。
- 编辑规则页。
- 系统配置页。
- SQL Server 数据源编辑页。
- 执行日志页。
- 邮件日志页。

页面设计要求：

- 以后台管理效率为主。
- 表单字段清晰。
- 错误提示明确。
- 列表信息便于扫描。
- 不展示敏感密码明文。

## 5. 非功能需求

### 5.1 安全性

- 所有管理页面必须登录后访问。
- 管理员密码使用哈希保存。
- SQL Server 密码和 SMTP 密码加密保存。
- 不在页面、接口响应和日志中输出密码明文。
- SQL 执行限制为只读查询。
- 生产环境数据库账号必须遵循最小权限原则。

### 5.2 可靠性

- SQL Server 连接失败不能导致 Web 服务崩溃。
- SMTP 发送失败必须记录日志。
- Worker 调度单条规则失败不应影响其他规则。
- 定时任务同一规则避免并发重复执行。

### 5.3 可维护性

- Web、Worker、执行器、SQL 客户端、邮件发送、模板渲染应保持模块边界清晰。
- 外部服务调用应通过适配器封装，便于测试和替换。
- 关键逻辑应有自动化测试覆盖。

### 5.4 部署要求

- 系统使用 Python / FastAPI 实现。
- 系统自身数据使用 SQLite 保存。
- 业务数据源为 SQL Server。
- 生产环境需要安装 Microsoft ODBC Driver for SQL Server。
- Web 服务和 Worker 可以独立启动。

## 7. 验收标准

系统达到以下条件视为第一版可验收：

1. 管理员可以登录系统。
2. 管理员可以配置 SQL Server 数据源。
3. 管理员可以配置 SMTP 邮件服务。
4. 管理员可以创建和编辑预警规则。
5. 规则 SQL 保存前会进行安全校验。
6. “检测 SQL”可以连接 SQL Server 做语法检测。
7. 管理员可以手动执行规则。
8. Worker 可以按 Cron 自动执行启用规则。
9. SQL 查询有结果时，系统可以发送预警邮件。
10. SQL 查询无结果时，系统记录成功但不发送邮件。
11. 执行失败时，系统记录失败原因。
12. 邮件发送失败时，系统记录邮件日志。
13. 管理员可以在页面查看执行记录和邮件发送日志。
14. 密码类敏感信息不会以明文展示。

## 8. 当前版本边界

第一版暂不包含以下能力：

- 多角色权限体系。
- SSO、LDAP、AD 登录。
- 规则审批流程。
- 规则版本历史。
- 重复预警抑制。
- 恢复通知。
- 按 SQL 字段动态决定收件人。
- 多数据库类型支持。
- 多 Worker 队列化执行。
- 图表化报表分析。

这些能力可作为后续版本规划。
